# Diagnosticare e risolvere errori comuni

## Scenario

Durante l'integrazione iniziale, durante il primo test end-to-end con app IO o nell'esercizio quotidiano, il Relying Party può imbattersi in errori che si manifestano come sintomi osservabili (un'app che non si apre, una POST che non arriva, un JWT che non si decifra) senza che il codice di errore sia immediatamente parlante.

Questa sezione raccoglie i sintomi più frequenti, le cause probabili e le azioni correttive. È organizzata per fase del flusso, dalla configurazione iniziale fino alla verifica della risposta.

## 5.4.1 Errori in fase di onboarding e Entity Configuration

| Sintomo                                                                                                                       | Causa probabile                                                                                                                          | Azione correttiva                                                                                                                                            |
| ----------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `GET /list` di IPZS non include l'Entity Identifier del RP                                                                    | L'onboarding non è stato completato, oppure IPZS ha registrato l'Entity Identifier con una grafia diversa (es. con o senza slash finale) | Verificare la risposta IPZS via PEC, confrontare l'Entity Identifier dichiarato con quello effettivamente registrato. Contattare IPZS in caso di discrepanza |
| `GET /.well-known/openid-federation` restituisce `Content-Type: application/jwt` invece di `application/entity-statement+jwt` | Configurazione errata del server HTTP del RP                                                                                             | Configurare esplicitamente il `Content-Type` corretto per la rotta `/.well-known/openid-federation`                                                          |
| `/resolve` del Trust Anchor risponde con `invalid_trust_chain`                                                                | `authority_hints` non valorizzato correttamente, oppure `x5c` mancante o non valido nel JWK Set                                          | Verificare che `authority_hints` contenga esclusivamente l'URL del Trust Anchor IPZS dell'ambiente in uso, e che ogni Federation Key abbia il proprio `x5c`  |
| Il Trust Mark è presente nell'EC ma `/resolve` continua a fallire                                                             | Il Trust Mark è scaduto (claim `exp` nel passato)                                                                                        | Richiedere un nuovo Trust Mark a IPZS e aggiornare l'EC                                                                                                      |
| L'EC viene ripubblicato ma le altre entità continuano a leggere la versione precedente                                        | Cache HTTP intermedio (CDN, proxy)                                                                                                       | Configurare `Cache-Control: no-cache` o impostare un TTL di pochi minuti sulla rotta dell'EC                                                                 |

## 5.4.2 Errori in fase di indirizzamento al Wallet

| Sintomo                                                                                                            | Causa probabile                                                                                                                                   | Azione correttiva                                                                                                                                                              |
| ------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| L'app Wallet non si apre dopo la scansione del QR Code                                                             | L'`authorization_endpoint` della Wallet Solution selezionata non corrisponde a un Universal Link registrato sul sistema operativo del dispositivo | Verificare i metadata della Wallet Solution recuperati dal Registry; in caso di assenza di un Universal Link valido, utilizzare lo schema custom `openid4vp://` o `haip-vp://` |
| L'app si apre ma mostra un errore generico ("Richiesta non valida")                                                | Il `client_id` non è prefissato con `openid_federation:`, oppure non coincide con l'`iss` registrato                                              | Verificare il payload del Request Object: `client_id` deve essere `openid_federation:<entity_identifier>`                                                                      |
| L'app si apre ma non mostra il logo del Relying Party                                                              | `logo_uri` non è un file SVG, oppure non è raggiungibile da rete mobile (dietro firewall, CDN protetta)                                           | Verificare il MIME type del logo (`image/svg+xml`) e la raggiungibilità da reti pubbliche                                                                                      |
| Il nome del Relying Party visualizzato nell'app è generico (es. "Servizio web") anziché quello dell'organizzazione | `client_name` non valorizzato nei metadati `openid_credential_verifier`                                                                           | Aggiungere o correggere `client_name` nel blocco `openid_credential_verifier` dell'EC                                                                                          |

## 5.4.3 Errori in fase di ricezione della risposta

| Sintomo                                             | Causa probabile                                                                                                                                       | Azione correttiva                                                                                                                                                  |
| --------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Il `response_uri` non riceve alcuna POST            | L'URL utilizzato nel Request Object non coincide con uno degli URL registrati in `response_uris` nell'EC                                              | Verificare la coerenza fra il valore di `response_uri` nel Request Object e il contenuto di `response_uris` nell'EC; verificare che l'EC pubblicato sia aggiornato |
| Il `response_uri` riceve la POST ma il body è vuoto | Il client HTTP del Wallet non sta inviando il `Content-Type: application/x-www-form-urlencoded`, oppure il proxy davanti al RP sta rimuovendo il body | Loggare l'intera richiesta HTTP in arrivo, verificare la presenza del parametro `response=<JWE>`                                                                   |
| Il JWE non si decifra (errore di crittografia)      | La Protocol Key utilizzata dal Wallet per cifrare non corrisponde a quella esposta in `openid_credential_verifier.jwks` dell'EC                       | Verificare che la Protocol Key nell'EC sia aggiornata; verificare che la chiave privata corrispondente sia quella correttamente utilizzata in decifratura          |
| `state` decifrato non coincide con la sessione      | Sessione scaduta lato RP, oppure attacco CSRF in corso                                                                                                | Restituire `400 Bad Request` al Wallet; verificare i timeout di sessione lato RP                                                                                   |

## 5.4.4 Errori in fase di verifica della credenziale

| Sintomo                                                        | Causa probabile                                                                                                                                              | Azione correttiva                                                                                                                                 |
| -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| La Trust Chain dell'issuer non si risolve                      | L'EC dell'issuer non è raggiungibile, oppure il Subordinate Statement che lo certifica è scaduto                                                             | Verificare manualmente `GET <issuer>/.well-known/openid-federation`; chiamare `GET /resolve?sub=<issuer>` del Trust Anchor per validare la catena |
| La firma dell'Issuer-Signed-JWT non risulta valida             | La chiave pubblica utilizzata per la verifica non è quella corretta dell'issuer (es. estratta dalla cache invece che dalla Trust Chain freschamente risolta) | Recuperare la chiave pubblica dall'`x5c` dell'header del SD-JWT, oppure dalla Trust Chain risolta in tempo reale                                  |
| Una o più Disclosure falliscono la verifica `hash in _sd`      | Le Disclosure non sono codificate correttamente in base64url, oppure è stata corrotta la stringa di Combined Presentation in fase di split sul `~`           | Validare manualmente con [sdjwt.org](https://sdjwt.org/) per identificare la Disclosure errata                                                    |
| Il KB-JWT fallisce la verifica della firma                     | La chiave utilizzata per la verifica non è `cnf.jwk` dell'Issuer-Signed-JWT, oppure il Wallet ha firmato con una chiave diversa                              | Estrarre `cnf.jwk` dal payload dell'Issuer-Signed-JWT (non dall'header) e utilizzarla esclusivamente per il KB-JWT                                |
| Il KB-JWT è firmato correttamente ma `sd_hash` non corrisponde | Le Disclosure utilizzate per il calcolo del `sd_hash` non corrispondono a quelle effettivamente inviate, oppure manca il `~` finale nel calcolo              | Calcolare `sd_hash = base64url( SHA-256( issuer_jwt + "~" + join(disclosures, "~") + "~" ) )` con il `~` finale presente                          |
| Il KB-JWT è valido ma `nonce` non coincide                     | Il `nonce` salvato in sessione lato RP è stato sovrascritto da una sessione successiva                                                                       | Garantire che la chiave di sessione lato RP sia unica per ciascuna presentazione                                                                  |
| La Status List indica credenziale revocata                     | La credenziale è effettivamente revocata dal Credential Issuer                                                                                               | Rifiutare la presentazione; informare l'utente che la credenziale presentata non è più valida                                                     |

## 5.4.5 Strumenti di diagnosi

| Strumento                                     | Uso tipico in diagnosi                                                                                                   |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| [jwt.io](https://jwt.io/)                     | Decodificare un JWT (header e payload) e verificarne la firma con la chiave pubblica corrispondente                      |
| [sdjwt.org](https://sdjwt.org/)               | Analizzare una SD-JWT Combined Presentation: visualizzare l'Issuer-Signed-JWT, le Disclosure decodificate e il KB-JWT    |
| [CyberChef](https://gchq.github.io/CyberChef) | Codificare e decodificare base64url, calcolare SHA-256, manipolare stringhe binarie                                      |
| `openssl x509 -inform DER -noout -text`       | Decodificare un certificato X.509 estratto da un campo `x5c` per ispezionare subject, issuer, scadenza                   |
| `curl -v`                                     | Tracciare l'intera transazione HTTP fra Relying Party e Trust Anchor (oppure fra Wallet e Relying Party in fase di test) |
| `wallet-conformance-test`                     | Riprodurre uno scenario di errore in ambiente controllato per isolare il problema                                        |
